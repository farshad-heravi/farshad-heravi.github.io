Project Overview

The Goal

Make a real OnRobot RG6 industrial gripper a first-class citizen of the ROS 2 ecosystem — controllable through the exact same ros2_control interfaces, action servers, and MoveIt 2 planning pipeline used for the robot arm it sits on, with no bespoke node or custom protocol leaking into application code. The deliverable is a clean hardware interface plugin that an integrator can drop into any robot's URDF and launch, turning vendor-specific Modbus registers into a standard prismatic joint with position command and position/velocity state.

Key Highlights

  • Standards-compliant ros2_control plugin — a SystemInterface exported through pluginlib (onrobot_rg6_hardware/Rg6SystemInterface), so the gripper is driven by the stock GripperActionController and JointStateBroadcaster rather than any custom interface.
  • Real industrial-hardware integration over Modbus/TCP to the OnRobot Compute Box — decoding the vendor register map (force, width, control word, status) into clean joint-space semantics.
  • RAII C++ design — a ModbusClient wrapper owns the libmodbus connection lifecycle, paired with a fully implemented hardware lifecycle (init → configure → activate → deactivate → cleanup) and explicit error returns on communication faults.
  • MoveIt 2-ready kinematics — a reusable Xacro macro models the RG6's four-bar linkage with finger/knuckle mimic joints, so the gripper visualizes and plans correctly inside a larger robot model.
  • Robust state mapping — current opening is read each cycle, corrected for finger width into the inner-finger-gap convention, and velocity is estimated from successive position samples.
  • Single-command launch — one launch file brings up robot_state_publisher, the controller manager at 50 Hz, and both controllers, parameterized by IP, port, slave ID, and target force.

Technical Deep Dive

The read() / write() Cycle

The controller manager ticks the interface at 50 Hz. On each write(), the target force, the commanded width (joint command plus finger width), and a control word are packed into the gripper's holding registers via Modbus FC16 (write multiple registers) at base 0x0000 — with redundant writes suppressed when the command is unchanged. On each read(), a ten-register block is fetched via FC3 (read holding registers) from 0x0102, yielding the status word and actual opening, from which the joint position and an estimated velocity are derived.

Register Map & Joint Semantics

Vendor registers speak in tenths of a millimetre and tenths of a Newton; the application speaks in metres of inner finger gap. The interface owns that translation: hardware width = commanded position + finger_width, force scaled into the 25–120 N range, and status bits (busy, grip-detected) surfaced from the read block. Centralizing this mapping keeps every downstream consumer — controllers, MoveIt, user code — in clean SI joint space.

Lifecycle & RAII Connection Management

The connection follows the ros2_control lifecycle precisely: on_init parses and validates URDF hardware parameters; on_configure opens the Modbus/TCP socket and seeds state from the live hardware; on_activate starts the loop; on_deactivate issues a stop; on_cleanup closes the socket. The RAII ModbusClient guarantees the connection is released even on faults, and communication failures are propagated as hardware-interface errors rather than silently swallowed.

MoveIt 2 Kinematic Model

A single prismatic gripper_width_joint drives the device, but the visual/collision model is the full four-bar linkage with finger and knuckle mimic joints expressed in the Xacro macro. This lets MoveIt 2 plan and collision-check against an accurate gripper geometry while the actuator abstraction stays a one-DoF width command — and the prefix-parameterized macro drops into any multi-robot URDF.

Usage at a Glance

Bring-up is a single launch command; commanding the gripper is a standard GripperCommand action — the same call that works for any ROS 2 gripper, now backed by real hardware.

# Launch the driver + controllers
ros2 launch onrobot_rg6_hardware rg6_control.launch.py \
  ip:=192.168.1.1 port:=502 slave_id:=65 target_force:=40

# Close the gripper (inner finger gap = 2 mm)
ros2 action send_goal /gripper_controller/gripper_cmd \
  control_msgs/action/GripperCommand \
  "{command: {position: 0.002, max_effort: 40.0}}"

# Open fully (152 mm)
ros2 action send_goal /gripper_controller/gripper_cmd \
  control_msgs/action/GripperCommand \
  "{command: {position: 0.152, max_effort: 40.0}}"

Engineering Notes & Honest Trade-offs

A driver that touches real hardware is only as trustworthy as its documented edges. A few are worth calling out explicitly:

  • Runtime force. Target force is currently fixed by the launch/URDF parameter; the per-goal max_effort field of the action is not yet forwarded to the force register at runtime — a deliberate, clearly-scoped next step rather than a hidden gap.
  • Register provenance. The read-block offsets were validated against a working device and a community implementation; they are flagged in the docs as warranting a cross-check with OnRobot's official Modbus specification before safety-critical deployment.
  • Connection exclusivity. The Compute Box permits a single active Modbus/TCP connection, so the interface is built around owning exactly one — reflected directly in the RAII connection design.
