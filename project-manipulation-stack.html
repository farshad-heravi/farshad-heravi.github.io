Project Overview

The Goal

Build a clean, reusable manipulation stack that takes a simulated robotic arm from a high-level pick-and-place request all the way down to collision-free, physically executed motion — without coupling the user to a single robot, gripper, or scene layout. The library is engineered so that the computationally heavy components (inverse kinematics, sampling-based planning, trajectory generation, torque control) live in a performant C++ engine, while scene composition and orchestration stay ergonomic in Python. The two layers are bridged with PyBind11, giving a single-import Python API backed by compiled C++ internals.

Key Highlights

  • Two-layer architecture — a compiled C++17 engine for IK, planning, trajectories and control, wrapped by a Python orchestration and scene-composition layer through PyBind11.
  • Robot- and gripper-agnostic — runtime-selectable across UR5e, Franka Panda, and xArm7 arms with Robotiq 2F-85 grippers, assembled dynamically rather than baked into a fixed model.
  • Collision-aware RRT planner built directly on MuJoCo's contact queries, with grasp-aware collision filtering so a held object moves with the gripper during planning.
  • Smooth, limit-respecting execution via quintic polynomial trajectory generation feeding a per-joint PD torque controller stepped through MuJoCo's physics loop.
  • Real-time observability — full 3-D telemetry streamed live to the Rerun viewer for inspection and debugging.
  • Modern, reproducible build — CMake + scikit-build-core packaging with a single pip install -e . producing the compiled extension.

System Architecture

A Two-Layer Design

The stack is deliberately split into two cooperating layers. The C++ engine layer (src/, include/pnp/) contains the performance-critical numerical core: ik.cpp (Damped Least-Squares IK), planner.cpp (RRT with collision detection), trajectory.cpp (quintic polynomial trajectory generation), and controller.cpp (PD torque control). The Python layer (python/pnp/) handles everything that benefits from flexibility and rapid iteration: scene.py composes the MJCF scene at runtime, system.py exposes the high-level PnpSystem orchestrator, and visualizer.py streams telemetry to Rerun. PyBind11 bindings in bindings/pypackage.cpp stitch the layers together so that calling a planner from Python invokes compiled C++ with no glue code on the user's side.

This separation means the heavy inner loops — Jacobian assembly, nearest-neighbour search, per-step collision checks, torque integration — run at compiled speed, while scene setup and task logic stay readable and quick to modify. Either layer can be reworked independently behind the binding boundary.

The Pick-and-Place Pipeline

A task flows through six well-defined stages: (1) Scene Construction — robot, gripper, table, tray and objects are assembled into a single MJCF model; (2) IK Solving — the target end-effector pose is converted into a joint-space goal; (3) Motion Planning — RRT searches for a collision-free waypoint sequence; (4) Trajectory Smoothing — waypoints are interpolated with quintic polynomials respecting velocity limits; (5) Physics Execution — the trajectory drives MuJoCo's actuators through the mj_step loop while gripper state is managed; and (6) Visualization — the evolving 3-D state is streamed to Rerun. The bundled ExampleTaskManager chains these into a full sequence: approach → descend → grip → lift → transport → drop → retreat.

Key Algorithms

Inverse Kinematics — Damped Least-Squares

The IK solver builds the 6×nv Jacobian using MuJoCo's mj_jacBody / mj_jacSite and solves the damped normal equations (JᵀJ + λ²I) Δq = Jᵀe iteratively, refining the joint configuration while clamping to joint limits at every step. The damping term λ keeps the solver well-conditioned near singularities, trading a little tracking accuracy for numerical stability where an undamped pseudo-inverse would blow up.

Motion Planning — Grasp-Aware RRT

Planning runs as a Rapidly-exploring Random Tree in joint space with a 10% goal bias, querying MuJoCo's contact engine for validity. The collision logic is context-sensitive rather than naïve: environment-to-environment contacts are ignored, robot-to-object contacts are skipped during the approach phase, and — crucially — when the gripper is holding an object, the object's pose is snapped to the gripper before every collision check. This lets the planner reason about a grasped payload as part of the robot, avoiding the classic failure where a held object phantom-collides with the world or is ignored entirely.

Trajectory Smoothing — Quintic Polynomials

Raw RRT waypoints are jagged and time-unaware. The trajectory generator fits quintic (5th-order) polynomials between them, producing C²-continuous motion with smooth position, velocity and acceleration profiles while honouring per-joint velocity limits — the difference between a path that exists and a trajectory a controller can actually track.

Control — PD Torque Law

Execution closes the loop with a per-joint PD torque controller, τᵢ = k_p·(q_target[i] − q[i]) + k_d·(q̇_target[i] − q̇[i]), computed each tick and applied to MuJoCo's actuators inside the mj_step integration loop. This drives the arm along the smoothed trajectory in a physically consistent simulation rather than teleporting joints to setpoints.

Scene Composition & Mesh Handling

Dynamic MJCF Assembly

Rather than shipping a fixed world model, the PySceneManager composes scenes at runtime through the dm_control MJCF API — selecting a robot (UR5e, Panda, or xArm7), attaching a gripper (Robotiq 2F-85), and placing a table, tray and arbitrary objects. The same orchestrator (PnpSystem) therefore works across hardware without code changes; the robot and end-effector are simply parameters: pnp.setup(robot_type="ur5e", ee_type="robotiq_2f85").

Robust STL Ingestion

Custom objects loaded from STL files are processed automatically so the simulation stays performant and valid: ASCII meshes are detected and converted to binary, meshes exceeding 200,000 faces are decimated with Open3D, and an axis-aligned bounding box is computed to generate an invisible collision proxy. The visual mesh stays high-fidelity while the planner reasons against a cheap, reliable collision surrogate.

Python API at a Glance

The whole stack collapses into a handful of high-level calls — robot selection, motion, and gripper control are one-liners, with the compiled C++ engine doing the heavy lifting underneath.

from pnp import PnpSystem

pnp = PnpSystem(use_rerun=True)
pnp.setup(robot_type="ur5e", ee_type="robotiq_2f85")

pnp.move_to_pose([0.5, 0.2, 0.3])   # IK + collision-free RRT + smoothed execution
pnp.close_gripper(target_val=200.0) # grasp
pnp.open_gripper()                  # release
